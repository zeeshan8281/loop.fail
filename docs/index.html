How to play

The loop CLI drives the whole flow — pick a problem, scaffold a squad, run, submit.

# 1 · set up
git clone <your fork> && cd loop.fail
pip install -r requirements.txt
./loop config --handle <your_github_handle>
export DARKBLOOM_API_KEY=...        # and/or ANTHROPIC_API_KEY / OPENAI_API_KEY

# 2 · pick a problem & scaffold a squad
./loop tasks
./loop start 001 --handle <you>    # copies a starter loop into submissions/<you>.py

# 3 · run it (bounded loop, temperature=0, kill-switch caps)
./loop run 001                    # add --mock for a keyless dry run

# 4 · submit a solved run → get the PR commands
./loop submit 001
size

3 · 5 · 7 agents

Squad size is penalized. Start at 3; grow only if 3 can't crack it.

cost

Cheap models win

Cost dominates the score. Keep Darkbloom on the heavy lifting; a strong model only where reasoning is needed.

loop

Bound every loop

The runner enforces each task's iteration, token, and wallclock caps — the kill switch.

Leaderboards

One board per problem · lowest score wins · 🏆 marks the current leader.

The four architectures

Pick one — you can't define your own. The kill switch (iteration/token/wallclock caps) is built in.

sequential seq

Agents run in order; each receives the previous one's output. Simplest — good for tight fixes.

debate deb

Everyone answers, sees the others, revises. A moderator picks the final. Good for harder reasoning.

supervisor sup

The canonical Session 5 pattern. A supervisor verifies the workers; unverified → run again with feedback.

manager mgr

One planner assigns subtasks to the workers, collects results, and synthesizes the final answer.
